# DE-002 — Design Data Security for Engineering Pipelines

**Version**: 2.0
**Domain**: data-engineering
**Level**: ADVANCED
**FORGE**: PASS

## Problema
Data breaches in pipelines cost an average of $4.45M per incident (IBM 2024). Security bolted on after deployment creates gaps at every handoff — ingestion, storage, processing, and serving — that attackers exploit through misconfigured IAM, unencrypted data in transit, or over-permissioned service accounts.

## Procedura
### Prerequisites
- Cloud platform access (AWS/GCP/Azure) with IAM admin rights
- Understanding of data classification tiers (public/internal/confidential/restricted)
- Access to a secrets manager (HashiCorp Vault, AWS Secrets Manager, or GCP Secret Manager)

### Steps
1. **Classify data assets by sensitivity** — Scan all data stores with tools like AWS Macie, Google DLP, or open-source Presidio. Tag every column/field as: PII (names, emails, SSN), PHI (health records), PCI (payment data), or internal. Benchmark: classification should cover 100% of production tables within 2 weeks for a mid-size org.

2. **Implement encryption at rest and in transit** — Enable AES-256 encryption for all storage (S3 SSE-KMS, BigQuery default encryption, Snowflake Tri-Secret Secure). Enforce TLS 1.3 for all data in transit. For Kafka: enable SSL listeners and SASL authentication. Use customer-managed keys (CMK) for confidential/restricted data — rotate keys every 90 days.

3. **Design least-privilege IAM for pipeline components** — Create dedicated service accounts per pipeline (not shared). In Snowflake: use roles hierarchy (SYSADMIN → ANALYST_ROLE → READER_ROLE). In BigQuery: use dataset-level IAM with `roles/bigquery.dataViewer` for consumers, `roles/bigquery.dataEditor` for ETL. In Airflow: use connection-level credentials, never environment variables. Audit: zero service accounts should have admin/owner roles.

4. **Implement dynamic data masking and tokenization** — Configure Snowflake Dynamic Data Masking policies for PII columns (mask SSN to `XXX-XX-1234`, email to `j***@domain.com`). For BigQuery: use policy tags with column-level security. For non-production environments: use Tonic.ai or Delphix to generate synthetic data. Benchmark: masking should add <5ms per query.

5. **Set up pipeline-level access control** — In Airflow: implement DAG-level access control with RBAC (Admin/User/Viewer roles). In dbt: use grants in `dbt_project.yml` to auto-assign permissions on model creation. For Kafka: configure ACLs per topic (producer/consumer separation). For Fivetran/Airbyte: use workspace-level permissions with SSO integration.

6. **Configure comprehensive audit logging** — Enable CloudTrail (AWS), Audit Logs (GCP), or Activity Log (Azure) for all data services. In Snowflake: query `SNOWFLAKE.ACCOUNT_USAGE.ACCESS_HISTORY` for column-level access tracking. In BigQuery: enable Data Access audit logs. Set retention to 365 days minimum. Stream audit logs to SIEM (Splunk/Datadog) for real-time alerting.

7. **Implement network security boundaries** — Deploy data infrastructure in private subnets with no public IPs. Use VPC Service Controls (GCP) or PrivateLink (AWS/Snowflake) for service-to-service communication. Configure firewall rules to allow only known IP ranges. For Kafka: deploy in private subnet with client VPN access only. Benchmark: zero data services should be publicly accessible.

8. **Build secret rotation automation** — Store all credentials in a secrets manager (never in code, config files, or environment variables). Automate rotation: database passwords every 30 days, API keys every 90 days, service account keys every 180 days. In Airflow: use `SecretsBackend` to pull from Vault/AWS Secrets Manager. Test rotation in staging before production.

9. **Implement data loss prevention (DLP) gates** — Add DLP scanning at ingestion points: scan files before loading to data lake, validate API responses for unexpected PII. In BigQuery: use Cloud DLP API to scan tables on schedule. Block pipeline execution if restricted data is detected in unauthorized zones. Alert security team within 15 minutes of any DLP violation.

10. **Run security validation and penetration testing** — Conduct quarterly security audits: IAM review (identify over-permissioned accounts), encryption verification (scan for unencrypted data stores), network scan (check for public endpoints). Use tools like ScoutSuite or Prowler for cloud security posture assessment. Benchmark: achieve 90%+ compliance score on CIS benchmarks.

### Verification
- All data stores have encryption at rest (AES-256) and in transit (TLS 1.3) verified
- Zero service accounts have admin/owner roles (least-privilege confirmed)
- Audit logs are streaming to SIEM with 365-day retention
- DLP scanning covers all ingestion points with <15min alert SLA
- Network scan confirms zero publicly accessible data services
