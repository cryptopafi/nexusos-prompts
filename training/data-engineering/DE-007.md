# DE-007 — Work with Python Data Structures for Data Engineering

**Version**: 2.0
**Domain**: data-engineering
**Level**: BEGINNER
**FORGE**: PASS

## Problema
Data engineers who default to pandas for everything waste memory and CPU — a 10M-element list comprehension is 5x faster than `df.apply()` for simple transforms. Mastering native Python data structures enables efficient data manipulation before reaching for heavy frameworks.

## Procedura
### Prerequisites
- Python 3.9+ installed
- Basic Python syntax knowledge

### Steps
1. **Master list operations for data pipelines** — Use list comprehensions for transforms: `[row['amount'] * 1.1 for row in records if row['status'] == 'active']` (3-5x faster than for-loop with append). Use `itertools.chain.from_iterable()` to flatten nested lists. For large datasets: use generator expressions `(x for x in data)` to avoid loading all into memory. Benchmark: list comprehension processes 1M items in ~100ms.

2. **Use dictionaries as in-memory lookup tables** — Build lookup dicts for reference data: `lookup = {row['id']: row for row in reference_data}`. Access is O(1) vs O(n) for list search. Use `collections.defaultdict(list)` for grouping: `groups[key].append(value)`. Use `dict.get(key, default)` to handle missing keys without try/except. Benchmark: dict lookup on 1M keys takes <1ms vs 500ms for list scan.

3. **Leverage sets for deduplication and membership testing** — Use sets for O(1) membership testing: `seen = set(); [x for x in data if x not in seen and not seen.add(x)]`. Find overlaps between datasets: `set_a & set_b` (intersection), `set_a - set_b` (difference). Benchmark: set membership on 1M elements takes <1ms; list `in` operator takes ~50ms.

4. **Use tuples and named tuples for immutable records** — Use `collections.namedtuple` or `typing.NamedTuple` for typed records: `Record = namedtuple('Record', ['id', 'name', 'amount'])`. Tuples use 20-30% less memory than dicts. Use as dict keys for composite lookups: `cache[(date, region)] = value`. For modern Python: use `@dataclass(frozen=True)` for immutable records with type hints.

5. **Implement efficient queue and stack patterns** — Use `collections.deque` for FIFO queues (O(1) append/popleft vs list's O(n) pop(0)). Implement sliding window calculations: `window = deque(maxlen=7)`. Use `heapq` for priority queues (top-N problems): `heapq.nlargest(10, records, key=lambda x: x['amount'])`. Benchmark: deque operations are 100x faster than list for queue patterns.

6. **Use Counter and OrderedDict for analytics** — `collections.Counter` for frequency analysis: `Counter(row['category'] for row in data).most_common(10)`. Merge counters with `+` operator. Use `Counter` to detect data skew before pipeline processing. For ordered operations: use `dict` (insertion-ordered since Python 3.7) or `OrderedDict` when explicit ordering semantics needed.

7. **Handle nested data structures (JSON/API responses)** — Write recursive flatteners for nested dicts: `flatten_dict(d, parent_key='', sep='.')`. Use `json.loads()` with custom decoders for dates/decimals. Parse nested API responses with safe navigation: `data.get('results', [{}])[0].get('value', None)`. For complex nesting: use `glom` library for declarative path extraction.

8. **Optimize memory with generators and itertools** — Use `yield` for streaming large files: process line-by-line without loading into memory. Chain transformations: `itertools.islice(filter(predicate, map(transform, data)), 1000)`. Use `itertools.groupby()` for sorted-data grouping (requires pre-sort). Benchmark: generator pipeline processes 100M rows in constant memory (~50MB) vs list approach (~8GB).

9. **Profile and choose the right structure** — Use `sys.getsizeof()` and `tracemalloc` to measure memory. Decision tree: need order + duplicates → list; need fast lookup → dict; need uniqueness → set; need immutability → tuple/frozenset; need FIFO → deque. Run `timeit` comparisons for hot paths. Document data structure choices in code comments for maintainability.

### Verification
- Can implement lookup table pattern with O(1) access confirmed via timeit
- Generator pipeline processes large file in constant memory (verified with tracemalloc)
- Nested JSON flattener handles 3+ levels of nesting correctly
- Data structure selection justified with benchmarks for at least 3 pipeline operations
