# DE-009 — Load and Manipulate Data with Pandas

**Version**: 2.0
**Domain**: data-engineering
**Level**: INTERMEDIATE
**FORGE**: PASS

## Problema
Pandas is the workhorse of Python data engineering, but naive usage leads to 10-100x slower pipelines — chained `.apply()` calls, copies on every operation, and string columns consuming 10x more memory than necessary. Production-grade pandas requires knowing the fast paths.

## Procedura
### Prerequisites
- Python 3.9+ with pandas 2.x, pyarrow backend
- Dataset with 100K-10M rows for benchmarking
- Jupyter notebook or IDE

### Steps
1. **Load data with optimized dtypes** — Always specify `dtype` dict: use `'int32'` instead of default `'int64'` (50% memory saving), `'category'` for columns with <1000 unique values (80-95% savings), `'string[pyarrow]'` for text (40% less memory than object). Use `pd.read_parquet()` over `pd.read_csv()` when possible (5-50x faster). Benchmark: 10M rows with optimized types uses 200MB vs 2GB with defaults.

2. **Master vectorized operations** — Replace `.apply()` with vectorized alternatives: `df['total'] = df['qty'] * df['price']` (100x faster than apply). Use `np.where()` for conditional logic: `np.where(df['amount'] > 1000, 'high', 'low')`. Use `.str` accessor for string operations: `df['name'].str.upper()`. Rule: if you write `apply(lambda`, there's almost always a vectorized alternative.

3. **Perform efficient groupby aggregations** — Use named aggregations: `df.groupby('region').agg(total=('amount', 'sum'), count=('id', 'count'), avg=('amount', 'mean'))`. For multiple stats: `.describe()` or custom `agg({'col': ['sum', 'mean', 'std']})`. Use `transform()` to add group-level stats back to rows without join. Benchmark: groupby on 10M rows with 1000 groups completes in <2s.

4. **Implement merge/join patterns** — Use `pd.merge()` with explicit `how`, `on`, `validate` parameters. Always validate: `validate='many_to_one'` prevents unexpected row multiplication. For large joins: ensure join keys are same dtype (int-to-int, not int-to-string). Use `indicator=True` to debug unmatched rows. Sort merge keys for merge-sort join optimization on sorted data.

5. **Handle time-series data** — Set datetime index: `df.set_index(pd.to_datetime(df['timestamp']))`. Resample: `df.resample('1H').agg({'value': 'mean', 'count': 'sum'})`. Rolling windows: `df.rolling('7D').mean()`. Handle timezones: `df.index.tz_localize('UTC').tz_convert('US/Eastern')`. Fill gaps: `df.asfreq('1H', method='ffill')`. Benchmark: rolling on 1M rows completes in <500ms.

6. **Chain operations with method chaining** — Write readable pipelines: `(df.pipe(clean_names).query('amount > 0').assign(tax=lambda x: x['amount'] * 0.2).groupby('category').agg(total=('amount', 'sum')))`. Use `.pipe()` for custom functions. Avoid intermediate variables that create memory copies. Use `inplace=False` (default) for chainability.

7. **Use pandas 2.x Arrow backend** — Enable PyArrow backend: `pd.options.mode.dtype_backend = 'pyarrow'` or `pd.read_parquet(path, dtype_backend='pyarrow')`. Benefits: native null support (no more float64 for nullable int), 2-3x faster string operations, direct Parquet/Arrow interop without copies. Benchmark: Arrow-backed string operations are 3-5x faster than object dtype.

8. **Optimize memory for large datasets** — Use `df.info(memory_usage='deep')` to profile. Downcast numerics: `pd.to_numeric(df['col'], downcast='integer')`. Process in chunks for >5GB: `for chunk in pd.read_csv(path, chunksize=500_000): process(chunk)`. When pandas isn't enough: switch to `polars` (2-5x faster, Rust-based) or `dask` (parallel, out-of-core). Decision point: >10GB single machine → polars; >100GB → Spark.

9. **Export data for pipeline consumption** — Write to Parquet (default for pipeline interchange): `df.to_parquet('output.parquet', index=False)`. Write to databases: `df.to_sql('table', engine, if_exists='append', method='multi', chunksize=10_000)`. For Snowflake: use `write_pandas()` from snowflake-connector (bulk COPY, 10-100x faster than INSERT). Generate quality report alongside data: row count, null counts, schema hash.

### Verification
- Vectorized operations used (zero `.apply(lambda` in production code)
- Memory optimized (>30% reduction from default types verified with `info(memory_usage='deep')`)
- Merge operations use `validate` parameter to prevent row multiplication
- Time-series operations produce correct results across timezone conversions
- Data export includes quality metadata (row count, schema hash)
