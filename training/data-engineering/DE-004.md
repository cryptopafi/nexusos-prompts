# DE-004 — Query and Analyze Data for Performance Tuning

**Version**: 2.0
**Domain**: data-engineering
**Level**: ADVANCED
**FORGE**: PASS

## Problema
Poorly performing queries cost organizations thousands in compute (a single full-table scan on a 1TB Snowflake table costs ~$2.30 per execution). Without systematic query analysis and tuning, data pipelines miss SLAs and analytics dashboards time out.

## Procedura
### Prerequisites
- Access to a database with production-scale data (PostgreSQL, MySQL, Snowflake, or BigQuery)
- Query profiling tools enabled (EXPLAIN, Query Profile)
- At least 1M+ rows in test tables

### Steps
1. **Baseline current query performance** — Run target queries and record: execution time (wall clock), rows scanned vs returned ratio, memory/CPU usage. In PostgreSQL: `EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)`. In Snowflake: check Query Profile for bytes scanned, partitions scanned vs total. In BigQuery: check bytes processed in job metadata. Benchmark: queries scanning >10x needed rows need optimization.

2. **Analyze execution plans for bottlenecks** — Identify anti-patterns: sequential scans on large tables (need index), nested loop joins on large datasets (switch to hash join), sort operations on unindexed columns, subquery materialization. In PostgreSQL: look for `Seq Scan` on tables >100K rows. In Snowflake: check for `TableScan` with low pruning efficiency (<50%).

3. **Optimize JOIN operations** — Rewrite joins: put the smaller table first in hash joins, replace correlated subqueries with JOINs (10-100x faster), use `EXISTS` instead of `IN` for large subquery results. In Snowflake/BigQuery: leverage automatic join optimization but verify with query profile. Benchmark: a well-optimized join should process <2x the output row count.

4. **Implement strategic indexing** — Create indexes for: WHERE clause columns (B-tree for equality/range), JOIN columns (composite index matching join order), ORDER BY columns. In PostgreSQL: use partial indexes (`WHERE status = 'active'`) for filtered queries, GIN indexes for JSONB/array columns. Monitor with `pg_stat_user_indexes` — drop indexes with <10 scans/month.

5. **Optimize aggregation and window functions** — Pre-aggregate in staging tables for repeated dashboard queries. Replace `COUNT(DISTINCT col)` with `APPROX_COUNT_DISTINCT()` (Snowflake/BigQuery) when exact count unnecessary — 50-90% faster. Use `QUALIFY` (Snowflake) instead of subquery for window function filtering. Partition window functions by high-cardinality columns.

6. **Tune warehouse/cluster sizing** — In Snowflake: right-size warehouse (XS for <10s queries, M for 10-60s, L for >60s). Enable auto-suspend (60s for dev, 300s for prod). In BigQuery: use BI Engine for sub-second dashboard queries (reserves 1-100GB RAM). In Redshift: run `STL_WLM_QUERY` to check queue wait times — add concurrency scaling if wait >5s.

7. **Implement query result caching** — In Snowflake: leverage automatic 24h result cache (same query, same warehouse). In BigQuery: cached results are free and automatic for 24h. For custom caching: materialize frequent queries as tables refreshed on schedule. In PostgreSQL: use materialized views with `REFRESH MATERIALIZED VIEW CONCURRENTLY`.

8. **Design partition and clustering strategies** — Partition tables by date (most common: daily/monthly). In BigQuery: use partitioned tables (save 50-90% scan cost). In Snowflake: define cluster keys on frequently filtered columns (date + region). In PostgreSQL: use declarative partitioning (`PARTITION BY RANGE`). Benchmark: partition pruning should reduce bytes scanned by >70%.

9. **Build a query performance monitoring dashboard** — Track: p50/p95/p99 query latency, bytes scanned per query, warehouse credit consumption (Snowflake), slot utilization (BigQuery). Set alerts for: query time >5x baseline, bytes scanned >2x expected, warehouse queuing >10s. Use Datadog, Grafana, or native monitoring (Snowflake Account Usage, BigQuery INFORMATION_SCHEMA).

10. **Document optimization playbook** — Record each optimization with: before/after metrics, technique applied, trade-offs (e.g., index speeds reads but slows writes by ~10%). Maintain a query performance registry for critical pipeline queries. Review quarterly and re-baseline after schema changes.

### Verification
- Target queries show >50% improvement in execution time or bytes scanned
- All production queries have execution plans reviewed (no full-table scans on >1M rows)
- Monitoring dashboard tracks p50/p95 latency with alerting configured
- Optimization playbook documents at least 5 specific tuning actions with before/after metrics
