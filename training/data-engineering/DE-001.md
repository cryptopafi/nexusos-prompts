# DE-001 — Architect a Data Platform (Layers, Components, Selection)

**Version**: 2.0
**Domain**: data-engineering
**Level**: ADVANCED
**FORGE**: PASS

## Problema
Organizations waste 6-18 months building fragmented data platforms because they skip architectural planning — resulting in redundant tools, ungoverned data silos, and 3-5x cost overruns when re-platforming.

## Procedura
### Prerequisites
- Cloud account (AWS/GCP/Azure) with admin access
- Business requirements document with data sources inventory
- Familiarity with at least one data warehouse (Snowflake, BigQuery, or Redshift)

### Steps
1. **Audit existing data landscape** — Catalog all data sources (databases, APIs, files, streams) using tools like Atlan, DataHub, or a manual inventory spreadsheet. Record volume (GB/TB), velocity (batch/real-time), variety (structured/semi/unstructured), and current latency SLAs. Benchmark: a mid-size org typically has 15-50 data sources.

2. **Define platform layers with technology mapping** — Design five layers: (1) Ingestion (Fivetran for SaaS connectors at 5-min sync, Airbyte for custom sources, Kafka for streaming at <100ms latency), (2) Storage (S3/GCS data lake for raw, Delta Lake/Iceberg for lakehouse), (3) Processing (Spark for batch >1TB, dbt for transformations <1TB, Flink for streaming), (4) Serving (Snowflake/BigQuery for analytics, Redis/DynamoDB for low-latency serving), (5) Orchestration (Airflow/Dagster for workflow management).

3. **Build a technology selection matrix** — For each layer, score 3+ candidate tools on: cost per TB/month, latency (p50/p99), ecosystem integration, team expertise, vendor lock-in risk. Example: Snowflake ($23/credit, auto-scale) vs BigQuery ($5/TB queried, serverless) vs Redshift ($0.25/hr per node, requires sizing). Weight criteria by business priority.

4. **Design the data modeling strategy** — Choose between star schema (simpler queries, 3-5 dimension tables per fact), snowflake schema (normalized dimensions, less storage), or Data Vault 2.0 (audit-friendly, handles source changes). For lakehouse: use medallion architecture (bronze=raw, silver=cleaned, gold=business-ready). Define naming conventions, partitioning strategy (by date for time-series, by region for geo-data).

5. **Architect data flow with SLA tiers** — Map each use case to an SLA tier: Tier 1 (real-time, <1s, Kafka+Flink), Tier 2 (near-real-time, <15min, micro-batch Spark), Tier 3 (batch, <4hr, Airflow+dbt). Document data lineage from source to serving. Use tools like dbt docs or Marquez for lineage tracking.

6. **Design for scalability and cost governance** — Set auto-scaling policies (Snowflake warehouse auto-suspend after 5min idle, BigQuery slot reservations for predictable workloads). Implement cost monitoring with tags per team/project. Benchmark: well-architected platforms maintain <$10/TB/month for storage, <$5/TB for processing.

7. **Implement security and governance layer** — Define column-level access control (Snowflake dynamic data masking, BigQuery column-level security), row-level security for multi-tenant data, encryption at rest (AES-256) and in transit (TLS 1.3). Set up Unity Catalog (Databricks) or Purview (Azure) for centralized governance.

8. **Create architecture decision records (ADRs)** — Document each technology choice with: context, decision, consequences, and alternatives considered. Include a C4 architecture diagram (context, container, component levels). Store ADRs in version control alongside infrastructure-as-code.

9. **Validate with a proof-of-concept pipeline** — Build one end-to-end pipeline covering all layers: ingest 1M+ rows from a production source, transform through bronze→silver→gold, serve to a dashboard. Measure: ingestion throughput (rows/sec), transformation time, query latency (p50/p95), total cost. Compare against SLA targets.

10. **Establish operational runbooks** — Document monitoring setup (Datadog/Grafana dashboards for pipeline health, data freshness, cost), alerting thresholds (pipeline failure, data delay >2x SLA, cost spike >20%), and incident response procedures. Schedule quarterly architecture reviews.

### Verification
- Architecture diagram covers all 5 layers with specific technology choices justified
- Technology selection matrix has scored 3+ alternatives per layer
- PoC pipeline meets defined SLA targets for latency, throughput, and cost
- ADRs are version-controlled and reviewed by at least one peer
- Cost governance tags and auto-scaling policies are active
