# VC-010 — Configure Cursor Rules for Custom AI Behavior

## Problema
Without custom rules, Cursor AI generates code in its default style — often mismatched with your project's conventions, tech stack, and quality standards, requiring manual corrections on 30-50% of outputs.

## Procedura
### Prerequisites
- Cursor AI installed (VC-001)
- Understanding of your project's coding standards

### Steps
1. **Understand rules hierarchy** — Cursor loads rules in order: (a) Global rules (~/.cursorrules), (b) Project rules (.cursorrules in project root), (c) Directory rules (.cursorrules in subdirectories). Project rules override global; directory rules override project.
2. **Create a global rules file** — `~/.cursorrules` for preferences that apply everywhere. Example content: "Always use TypeScript strict mode. Prefer const over let. Use async/await over .then(). Add JSDoc comments to exported functions. Use 2-space indentation."
3. **Create project-specific rules** — `.cursorrules` in project root. Include: framework (React 19 / Next 15 / Express), state management (Zustand / Redux), styling (Tailwind / CSS modules), testing (Vitest / Jest), database (Prisma / Drizzle), and naming conventions.
4. **Add architectural rules** — include patterns: "Use repository pattern for data access. Controllers should only handle HTTP concerns. Business logic goes in services/. Follow SOLID principles. Maximum function length: 30 lines."
5. **Add error handling rules** — "Always use try-catch in async functions. Return typed error responses using Result<T, E> pattern. Log errors with structured context (userId, action, timestamp). Never swallow errors silently."
6. **Add security rules** — "Never hardcode secrets. Use environment variables via process.env. Validate all user input with Zod schemas. Sanitize HTML output. Use parameterized queries (never string concatenation for SQL)."
7. **Add testing rules** — "Write tests for every public function. Use describe/it blocks with descriptive names. Mock external dependencies. Test both happy path and error cases. Minimum 80% coverage for new code."
8. **Use rules for language/format** — "Respond in English. Use concise commit messages (imperative mood, <72 chars). Write documentation in Markdown. Use TODO comments for incomplete sections."
9. **Test rules effectiveness** — generate code before and after adding rules. Compare: does the AI follow your conventions now? Does it use the correct framework APIs? Does it include error handling? Iterate on rules until AI output matches expectations 90%+.
10. **Version control rules** — commit `.cursorrules` to git. This ensures all team members get the same AI behavior. Review and update rules quarterly or when major dependencies change.

### Verification
- [ ] Global rules load (test by asking AI to generate code — should follow global style)
- [ ] Project rules override global (test with a conflicting rule)
- [ ] AI generates code matching project conventions without explicit reminders
- [ ] Security and error handling rules produce correct patterns automatically
- [ ] Rules file is committed to version control
