# VC-006 — Add New Features to Existing Projects with AI

## Problema
Adding features to existing codebases with AI often breaks existing functionality because developers fail to provide sufficient codebase context, leading to 2-3x rework cycles.

## Procedura
### Prerequisites
- Existing project with working codebase
- Cursor AI configured (VC-001)
- Git initialized with clean working tree (commit before adding features)

### Steps
1. **Commit current state** — `git add -A && git commit -m "checkpoint: pre-feature"`. This creates a rollback point. If AI changes break things, `git diff` shows exactly what changed.
2. **Map the affected area** — before prompting AI, identify: which files will be touched? What imports/dependencies exist? What patterns does the codebase use (e.g., MVC, repository pattern, hooks)? Attach 2-3 representative files as context.
3. **Write a feature spec prompt** — include: what the feature does, which existing patterns to follow, what NOT to change. Example: "Add a /notifications endpoint to @server.ts following the same pattern as @users.controller.ts. Use the existing auth middleware. Do NOT modify the database schema — use the existing Notification model."
4. **Use @codebase for pattern detection** — prefix your prompt with `@codebase` to let Cursor index the full project. Ask: "What patterns does this codebase use for new endpoints?" before writing the feature. AI will detect and follow conventions.
5. **Generate in Composer Agent mode for multi-file features** — `Cmd+I` → Agent. For features touching 3+ files (route + controller + model + test), Agent mode maintains coherence across files better than individual Chat prompts.
6. **Review diffs file-by-file** — after AI generates changes, use `git diff` or Cursor's built-in diff view. Check: (a) no unintended changes to existing files, (b) new code follows existing conventions, (c) imports are correct.
7. **Run existing tests first** — before testing the new feature, run the existing test suite. Any failures = regression. Fix regressions before adding new test coverage.
8. **Add tests for the new feature** — prompt: "Write tests for the new /notifications endpoint following the patterns in @users.test.ts. Cover: success, unauthorized, invalid input, empty results."
9. **Integration test** — test the new feature in the context of the full app. Check: does it work with existing auth? Does it handle edge cases? Does it play well with existing middleware?
10. **Clean commit** — `git add` only the new/modified files (not AI artifacts). Write a descriptive commit: "feat: add notifications endpoint with pagination and auth".

### Verification
- [ ] Existing test suite passes (zero regressions)
- [ ] New feature works end-to-end
- [ ] New tests cover success and error paths
- [ ] Code follows existing project conventions
- [ ] Clean git diff (no unintended changes)
